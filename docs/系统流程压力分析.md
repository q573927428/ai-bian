## 二、各模块详细流程与时间间隔

### 🔄 **模块1: KLineSimpleSyncService (K线数据同步服务)**

#### 职责
- 从Binance API同步K线数据到本地文件
- 按不同周期独立同步
- 事件驱动通知IndicatorsHub数据更新

#### 时间间隔配置

| K线周期 | 同步间隔 | 说明 |
|---------|---------|------|
| 5m      | 每5分钟 + 20秒延迟 | 等待K线完成 |
| 15m     | 每15分钟 + 20秒延迟 | 等待K线完成 |
| 1h      | 每小时 + 20秒延迟 | 等待K线完成 |
| 4h      | 每4小时 + 20秒延迟 | 等待K线完成 |
| 1d      | 每天 + 20秒延迟 | 等待K线完成 |

#### 单次同步数据量

```
假设监控8个交易对，5个周期：

单次同步操作：
├─ 每个交易对 × 每个周期 = 8 × 5 = 40次API请求
├─ 每次API请求：最新1-2根K线（约1KB数据）
├─ 总数据量：约40KB
└─ 事件触发：klineUpdated事件
```

#### 压力评估 ⚪ **低压力**

- ✅ 数据量小：每次仅同步最新K线
- ✅ 频率可控：按周期自然间隔
- ✅ 并发控制：同一周期不会同时同步
- ⚠️ 注意：首次同步历史数据压力较大（约1000根/交易对/周期）

---

### 🔄 **模块2: IndicatorsHub (指标计算缓存中心)**

#### 职责
- 从本地文件读取K线数据
- 计算所有技术指标
- 统一缓存供所有策略共享
- 定时刷新数据

#### 时间间隔配置

| 操作 | 间隔 | 说明 |
|------|------|------|
| 定时刷新 | 默认60秒 | 检查K线文件更新 |
| 事件驱动 | 实时 | 响应klineUpdated事件 |
| K线TTL | 按周期 | 5m:5分钟, 1h:1小时等 |

#### 单次刷新数据量

```
场景：10个策略，每个8个交易对，50%重叠 = 40个唯一交易对

单次refreshAllData()操作：
├─ 检查文件更新时间：40交易对 × 5周期 = 200次文件mtime检查
├─ 重新加载K线（如有更新）：约5-10个交易对/周期
├─ 重新计算指标：40交易对 × 5周期 × 6指标 = 1,200次指标计算
├─ 持仓量API请求：40次（每个交易对1次）
└─ 内存占用：约4-6MB
```

#### 指标计算详情

| 指标 | 计算复杂度 | 数据量 |
|------|-----------|--------|
| EMA  | O(n) | 300根K线 |
| RSI  | O(n) | 300根K线 |
| MACD | O(n) | 300根K线 |
| ATR  | O(n) | 300根K线 |
| ADX  | O(n) | 300根K线 |

#### 压力评估 🟡 **中等压力**

- ✅ 本地计算：无网络延迟
- ✅ 共享缓存：避免重复计算
- ⚠️ 峰值计算：每分钟约1,200次指标计算
- ⚠️ 文件I/O：检查200个文件mtime
- 💡 优化：事件驱动更新，减少全量刷新

### 🔄 **模块3: StrategyEngine (策略执行引擎)**

#### 职责
- 管理多个策略的独立定时循环
- 每个策略按配置间隔扫描市场
- 调用AI分析
- 执行交易信号

#### 时间间隔配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 策略扫描间隔 | 180秒(3分钟) | 可配置 |
| 交易对遍历 | 顺序执行 | 每个策略独立 |

#### 单次策略扫描流程

```
单个策略扫描（假设8个交易对，3个周期，所有指标）：

[策略主循环] - 每180秒触发一次
    │
    ├─ 遍历8个交易对（顺序执行）
    │   │
    │   ├─ [交易对1]
    │   │   ├─ 检查仓位冲突（内存操作）
    │   │   ├─ getBatchIndicators() - 从缓存读取
    │   │   │   ├─ 3周期 × 5指标 = 15次缓存读取
    │   │   │   └─ EMA周期计算（如需要）
    │   │   ├─ fetchPrice() - Binance API（1次）
    │   │   ├─ AI分析（带缓存）
    │   │   │   ├─ 检查缓存（内存）
    │   │   │   └─ 如未命中：DeepSeek API调用
    │   │   ├─ 验证信号
    │   │   └─ 执行交易（如有信号）
    │   │
    │   ├─ [交易对2]
    │   │   └─ ...同上...
    │   │
    │   └─ [交易对8]
    │       └─ ...同上...
    │
    └─ 完成，等待下一次循环
```

#### 数据量详细计算

**场景：10个策略同时运行**

| 项目 | 计算公式 | 数值 |
|------|---------|------|
| 总交易对扫描 | 10策略 × 8交易对 | 80次/3分钟 |
| 唯一交易对（50%重叠） | - | 40个 |
| 指标缓存读取 | 80次 × 3周期 × 5指标 | 1,200次/3分钟 |
| fetchPrice API调用 | 80次（可优化去重） | 80次/3分钟 |
| AI API调用（缓存命中率75%） | 80次 × 25% | 20次/3分钟 |

#### 压力评估 🟠 **中高压力（主要瓶颈）**

- ⚠️ **AI API调用**：最大压力点（成本+延迟）
- ⚠️ **价格API调用**：80次/3分钟，建议去重
- ⚠️ **顺序执行**：如果80个交易对每个耗时2秒，总耗时160秒
- ✅ **指标读取**：从缓存读取，压力小
- ✅ **仓位检查**：内存操作，压力小

### 🔄 **模块4: MultiStrategyAIAnalyzer (AI分析器)**

#### 职责
- 调用DeepSeek API进行市场分析
- 智能缓存（相同数据不重复调用）
- 降级处理

#### 时间间隔配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| AI缓存TTL | 10分钟 | 缓存有效期 |
| API超时 | - | 依赖网络 |

#### 单次AI分析数据量

```
单次AI调用：
├─ 输入Token：约1,500 tokens（提示词+指标数据）
├─ 输出Token：约300 tokens（JSON响应）
├─ 总Token：约1,800 tokens/次
└─ 响应时间：约2-5秒（网络依赖）
```

#### 压力评估 🔴 **高压力（最关键点）**

- 🔴 **API成本**：按1,800 tokens/次，20次/3分钟 = $10-15/天
- 🔴 **网络延迟**：每次2-5秒，累计时间长
- ⚠️ **并发限制**：DeepSeek API有速率限制
- ✅ **缓存优化**：75%命中率可大幅降低压力

### 🔄 **模块5: PositionMonitor (持仓监控)**

#### 职责
- 监控现有持仓
- 检查止盈止损
- 定时更新持仓状态

#### 时间间隔配置

| 操作 | 间隔 |
|------|------|
| 持仓检查 | 默认10秒 |
| 价格更新 | 实时/定时 |

#### 数据量
```
假设有5个持仓：
├─ 每10秒检查5个持仓
├─ 每次获取5个交易对当前价格
└─ 计算止盈止损（本地计算）
```

#### 压力评估 ⚪ **低压力**

- ✅ 持仓数量有限
- ✅ 本地计算为主
- ✅ 价格API可复用缓存

## 六、总结

### ✅ **系统优点**

1. **良好的缓存架构** - 多层缓存减少重复计算
2. **数据共享** - IndicatorsHub单例避免资源浪费
3. **本地K线存储** - 减少API依赖，提高稳定性
4. **模块化设计** - 各模块职责清晰

### ⚠️ **需要关注**

1. **AI API是最大瓶颈** - 成本和延迟都需要控制
2. **策略扫描可能超时** - 180秒间隔可能不够处理80个交易对
3. **建议测试** - 在实际负载下监控各模块耗时
